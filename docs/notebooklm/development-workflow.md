# AI 協作開發流程說明

## 文件目的

這份文件整理的是本專案的實際開發方式，也就是我如何把需求整理、文件規劃、AI 協作、測試驗證、Code Review、本機驗證、雲端部署與需求關閉串成一套穩定流程。

這份文件適合用在以下情境：

- 提供給面試官了解我的開發方法
- 提供給 NotebookLM 生成「開發流程介紹」簡報或影片
- 作為後續專案複用的 AI 協作開發 SOP

## 流程核心觀念

我的開發方式不是直接把需求丟給 AI 寫程式，而是先做需求收斂、文件整理、方案規劃，再進入實作與驗證。  
AI 在這個流程中扮演的是協作工具，而不是完全取代判斷的角色。  
真正的重點是：需求要先清楚、方案要先可驗證、測試要能證明結果、上線前後要有明確檢查點。

## 整體流程總覽

1. 統整需求
2. 使用 Gemini 搭配 NotebookLM 整理需求文件
3. 生成 PRD 與規格文件，交給 Codex 進行專案規劃
4. 檢核方案，必要時補充 test case
5. 執行方案，進入開發
6. 檢核測試設計與測試結果
7. 對核心功能進行 Code Review
8. 本機執行與手動驗證
9. 調整 CI/CD 與 Azure 環境設定
10. 上版並進行線上測試
11. 確認需求完成並關閉

## Step 1. 統整需求

### 目標

先把需求本身講清楚，避免過早進入實作，導致方向錯誤或反覆重工。

### 我會做的事

- 收集需求來源
- 拆解功能目標、使用者角色與預期成果
- 區分哪些是必要需求，哪些是加分項或後續擴充
- 明確定義本次開發範圍與不做的部分

### 這一步的重要性

很多專案不是卡在寫不出程式，而是卡在需求本身不夠明確。  
如果一開始沒有把角色、流程、輸入輸出與驗收條件說清楚，後面交給任何 AI 或工程師都容易偏掉。

### 建議輸出物

- 需求摘要
- 功能清單
- 角色清單
- 驗收標準草稿
- 邊界與限制說明

## Step 2. 使用 Gemini 搭配 NotebookLM 整理需求文件

### 目標

把初步需求從「零散資訊」整理成「可讀、可討論、可追蹤」的文件。

### 工具分工

- `Gemini`
  適合協助快速重整需求、補齊條列、整理邏輯順序
- `NotebookLM`
  適合做多文件彙整、主題萃取、摘要整理與後續簡報生成

### 我會做的事

- 將需求對話、草稿、參考資料整理成一組基礎文件
- 釐清角色流程與功能依賴
- 把口語需求轉成正式文件語言
- 補上背景說明與情境脈絡

### 這一步的價值

這一步能把需求從「我心裡知道要做什麼」轉成「任何協作者都看得懂的文件」，是後續讓 Codex 進行規劃的基礎。

## Step 3. 生成 PRD 與規格文件，交給 Codex 進行專案規劃

### 目標

在實作前，先把需求文件提升成可以驅動開發的規格文件，再交由 Codex 用 Plan 模式產生實作方案。

### 我會準備的文件

- PRD
- 功能規格文件
- API 或資料流草稿
- 資料模型說明
- 非功能性需求，例如安全性、可維護性、部署方式

### Codex 在這一步負責什麼

- 根據 PRD 與規格文件進行專案規劃
- 使用 Plan 模式拆出可執行方案
- 整理技術選型、模組邊界、開發順序與風險點
- 提出實作前需要確認的事項

### 這一步的檢查重點

- 規劃有沒有對齊需求目標
- 模組切分是否合理
- 開發順序是否能降低風險
- 方案是否能在現有專案結構中落地

## Step 4. 檢核方案，必要時補充 Test Case

### 目標

在真正寫程式前，先確認方案本身正確，必要時用 test case 補強邊界條件。

### 我會檢查什麼

- 方案是否真的滿足需求
- 有沒有忽略角色權限與資料邊界
- 有沒有與既有架構衝突
- 有沒有未處理的例外情境或失敗流程

### 補充 test case 的時機

如果某個功能是核心流程、邏輯比較複雜，或我希望 AI 更準確理解預期行為，我會在這一步補 test case 給 AI 參考，例如：

- 正常流程
- 權限限制
- 邊界輸入
- 錯誤情境
- 回歸風險

### 這一步的價值

這一步可以明顯降低「方案看起來合理，但實作後不符合預期」的機率。

## Step 5. 執行方案，進入開發

### 目標

在需求與方案都清楚後，才正式開始實作。

### 我會做的事

- 將確認過的方案交給 Codex 執行
- 按照規劃順序完成功能開發
- 讓 AI 先完成重複性高、結構明確的工作
- 人工持續檢查關鍵邏輯與整體方向

### 這一步的原則

- 不跳過規劃直接寫
- 不只看功能能不能跑，也看是否符合原本規格
- 保持文件、程式與測試的一致性

## Step 6. 檢核測試設計與測試結果

### 目標

確認測試不是為了湊數，而是真的能證明功能行為符合預期。

### 我會檢查什麼

- 測試是否符合 `AAA` 結構
- 測試名稱是否清楚表達情境與預期結果
- 測試資料是否可重現
- 測試是否覆蓋核心流程與重要邊界
- 測試結果是否與需求一致

### AAA 的重點

- `Arrange`
  測試前置資料是否清楚
- `Act`
  實際操作是否聚焦單一行為
- `Assert`
  驗證結果是否明確且對應需求

### 我重視的不只是有沒有綠燈

即使測試全部通過，我也會檢查：

- 測試是不是寫得太寬鬆
- 是否只驗 happy path
- 是否缺少權限、失敗、邊界等情境

## Step 7. 對核心功能進行 Code Review

### 目標

確認核心功能不只「能跑」，而是程式品質、可讀性、風險控制都在合理範圍內。

### Review 重點

- 核心邏輯是否正確
- 是否有權限漏洞或資料邊界問題
- 命名與模組責任是否清楚
- 是否引入不必要複雜度
- 測試是否足以支撐此次變更
- 是否可能造成回歸風險

### 我特別會看哪類功能

- 認證授權
- 核心商業規則
- 資料寫入流程
- 報表與通知這類對外輸出功能

### 這一步的價值

AI 很適合加速產出，但核心功能仍需要人的判斷來檢查風險與品質。

## Step 8. 本機 Run 起來測試

### 目標

在進入部署前，先確認整個專案在本機是能實際跑起來、能操作、能驗證的。

### 我會做的事

- 啟動前後端或相關服務
- 檢查啟動流程、設定檔與資料初始化
- 手動走一次核心流程
- 驗證 Swagger、登入、主要功能與關鍵畫面

### 為什麼這一步不能省略

有些問題不會在單元測試裡出現，例如：

- 環境變數錯誤
- migration 問題
- API 啟動失敗
- 本機資料與測試資料落差
- 實際操作體驗不順

## Step 9. 調整 CI/CD 與 Azure 環境設定

### 目標

把本機可運作的成果，轉成可在雲端穩定部署的版本。

### 我會做的事

- 調整 CI/CD pipeline
- 確認 build、test、publish 流程
- 設定 Azure 環境變數與連線字串
- 檢查部署目標環境是否具備必要資源
- 確認 migration、啟動與 health check 是否正常

### 這一步常見檢查點

- App Service 設定是否正確
- Azure SQL 連線是否正常
- JWT、Mail、CORS 等設定是否分環境管理
- 部署後是否能正常啟動

## Step 10. 上版並進行線上測試

### 目標

確認系統不只在本機正常，也能在實際部署環境中正確運作。

### 我會做的事

- 部署到目標環境
- 驗證主要 API 或前端流程
- 檢查線上環境設定與資料連線
- 進行冒煙測試與核心流程測試
- 確認日誌、錯誤訊息與監控資訊

### 線上測試的重點

- 角色登入是否正常
- 核心流程是否與本機一致
- 權限與資料顯示是否正確
- 部署是否造成非預期回歸

## Step 11. 確認需求完成並關閉

### 目標

在功能上線且驗證沒問題後，正式完成這次需求。

### 我會做的事

- 對照原始需求與驗收條件
- 確認文件、程式、測試與部署狀態一致
- 記錄必要的補充事項或後續優化項目
- 關閉需求

### 關閉需求前的最後確認

- 功能符合原始目標
- 測試結果符合預期
- 線上驗證完成
- 沒有阻塞性問題
- 已知限制有被記錄

## 這套流程的補充強化點

除了原本的 11 個步驟，我會建議補強以下觀念，讓整體流程更完整。

### 補充一：每一步都要有輸出物

如果每一步只有做事但沒有留下文件或結果，後面很難追蹤。  
因此建議每一步都盡量有可回看的輸出，例如：

- 需求摘要
- PRD
- 規劃方案
- 測試結果
- Review 結論
- 部署紀錄

### 補充二：核心功能要先定義驗收條件

越是重要的功能，越應該在開發前就定義什麼叫做完成。  
這會讓 AI 規劃、測試設計與人工檢核都更準確。

### 補充三：AI 負責加速，人負責判斷

這套流程的核心不是完全自動化，而是把 AI 放在最能提高效率的地方：

- 文件整理
- 方案拆解
- 程式生成
- 測試草稿
- Review 輔助

而人需要負責的則是：

- 需求判斷
- 方案核可
- 測試驗證
- 風險控管
- 上線確認

### 補充四：本機驗證與線上驗證不能互相取代

本機驗證能找出開發階段問題，線上驗證能找出環境與部署問題。  
兩者都必要，不能只做其中一個。

## 流程的價值總結

這套流程的價值在於：

- 先整理需求，再進入實作
- 先有文件，再有方案
- 先檢核方案，再開始寫程式
- 先驗證測試，再做核心 review
- 先本機確認，再進部署與上線
- 最後再正式關閉需求

這讓整個開發過程更可控，也讓 AI 協作真正成為效率工具，而不是不受控的黑盒子。
