## 版本一：60 秒電梯式介紹

這是一套以 ASP.NET Core 8 開發的校車管理後端系統，主要處理學生每週搭車登記、教師車上點名、管理端路線管理、通知提醒與報表匯出。  
我把它設計成一個 backend-first 的展示專案，所以不只做出 API，還把角色權限、資料模型、Swagger 文件、seed data 與整合測試一起補齊。  
面試時我可以直接在 Swagger 中切換管理員、教師、家長與學生角色，快速展示這個系統如何支援真實校務流程。

## 版本二：3 到 5 分鐘快速簡報腳本

### 開場

這個專案的題目是校車管理系統後端。  
我想解決的不是單一功能，而是一條完整的校車營運流程，包含家長每週登記、教師車上點名，以及管理端的路線調度、公告通知與報表輸出。

### 問題與角色

這套系統主要有四種角色：管理員、教師、家長與學生。  
家長負責替孩子填寫每週搭車安排，教師負責實際車上點名，管理員負責路線、調度、通知與報表，而學生則能查看自己的搭乘資料。  
我刻意把角色差異做明確，因為這類系統的核心難點不是畫面，而是資料可見範圍與流程責任。

### 技術設計

在技術上，我使用 ASP.NET Core 8、Entity Framework Core、ASP.NET Identity 與 JWT。  
整體架構採模組化單體，分成 Api、Application、Domain、Infrastructure 與 Contracts。  
這樣的好處是，面試時可以很清楚說明控制器、商業規則、資料存取與 API 契約各自負責什麼。

### 產品流程亮點

我覺得最有代表性的流程有三段。  
第一段是家長每週登記，系統支援直接複製上一週設定。  
第二段是教師依當天登記資料建立點名 session，逐筆更新學生狀態。  
第三段是管理員處理特殊調度、發送通知並產出報表。  
這三段加起來，可以把校車管理從事前規劃到事中執行都串起來。

### Demo 方式

Demo 時我會直接從 Swagger 開始，因為這能最快展示 API 文件、授權方式與角色切換。  
我會先用管理員帳號登入，再依序切到教師、家長與學生，讓面試官看到同一個系統如何隨角色改變可見功能。

### 收尾

這個專案不只是功能列表，而是一個可以被展示、被測試、也能繼續擴充的後端系統。  
如果下一步要更產品化，我會補上前端管理介面、正式通知通道與更細的權限策略。

## 版本三：8 到 10 分鐘面試 Demo 腳本

### 1. 專案背景

我做這個專案時，想把它設計成一個真實學校可能會需要的後端系統。  
因為校車管理會牽涉到固定週期登記、臨時改線、點名、通知與報表，所以很適合拿來展示後端系統設計能力。

### 2. 先講架構

整個專案採分層設計。  
Api 層負責對外 endpoint 與 Swagger。  
Application 層負責服務介面與協調邏輯。  
Domain 層放核心模型與商業規則。  
Infrastructure 層實作資料存取、JWT、通知與背景工作。  
Contracts 則定義 request / response DTO。  
這樣面對多角色流程時，邏輯不會全部混在 controller 裡。

### 3. 再講登入設計

登入這部分我沒有用單純 email，而是改成 `account`。  
學生用學號、教師與管理員用工號、家長用手機號碼。  
這個設計的重點是讓 Demo 更貼近真實場景，而不是只做一個最簡單的登入表單。

### 4. 開始展示管理員流程

我會先登入管理員帳號 `E0001`。  
登入成功後，可以先呼叫 `/api/auth/me` 確認角色，再用 `/api/admin/lookups` 取得管理端需要的學生與教職員資料。  
接著我會示範建立 dispatch override，說明如何處理單次調度。  
然後我會示範建立 broadcast 與報表，最後下載 CSV，讓面試官看到這不是假流程。

### 5. 展示教師流程

接下來我會切到教師帳號 `T0001`。  
先看 `/api/auth/context`，再看 `/api/routes`，可以發現教師只看得到自己被指派的路線。  
之後我會建立一個 attendance session，系統會根據當天登記資料產生點名名單。  
這能很直觀地說明前面的家長登記資料，後面真的會被營運流程使用。

### 6. 展示家長流程

接著切到家長帳號 `0900-000-003`。  
我會展示家長如何查詢 `2026-03-16` 這週的搭車資料，然後示範更新與套用上一週。  
這裡的重點是，這個系統不是只讓家長看資料，而是讓家長管理週期性的搭車需求。

### 7. 展示學生流程

最後切到學生帳號 `S10001`。  
學生可以看到自己的資料摘要，但權限範圍相對收斂。  
透過這個角色可以補強「同一個平台支援不同角色，但資料可見性不同」這個設計重點。

### 8. 工程亮點收尾

收尾時我會補幾個技術點。  
第一，專案啟動時會自動 migration 並 seed demo data，所以可以快速展示。  
第二，Swagger 已支援 Bearer Token 驗證，操作成本很低。  
第三，測試不只測 domain，也有 application 與 API integration tests。  
這代表專案不只是看起來完整，而是有基本驗證能力。
